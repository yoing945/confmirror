## 编码行为准则

**前提：** 除非用户明确要求其他语言，所有输出（包括回答、代码注释、Git 提交信息等）优先使用中文。

**权衡：** 更偏向谨慎，琐碎任务可自行判断。

> 琐碎任务指单文件、无歧义、不涉及核心逻辑的修改。

### 1. 编码前先思考
不假设，不掩饰困惑。实现前：
- 明确写出假设；不确定则提问。
- 存在多种解释时先列出，不默默选择。
- 存在多个可行方案时列出选项及优劣，让用户决定，不自行选定。
- 有更简单的方法直接指出。
- 困惑时停下，说清困惑点并提问。

### 2. 简单优先
只写解决问题所需的最少代码，不做预设性扩展：
- 有公认标准库或最佳实践时优先使用，即使自写方案看起来更少代码。
- 不加超出需求的功能。
- 不为一次性代码做抽象。
- 不加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写额外错误处理。
- 写了 200 行但 50 行就够时，重写。

自检：一个资深工程师会认为太复杂吗？会，就继续简化。

### 3. 外科手术式修改
只改必须改的，只清理自己造成的问题：
- 可指出相邻代码、注释或格式问题，但不顺手优化。
- 不重构未坏部分。
- 保持现有风格。
- 可指出无关死代码，但不删除。

因你的改动产生遗留项时，删除那些变成未使用的 import、变量或函数。

### 4. 目标驱动执行
将任务变更为可验证的目标：
- "添加校验" → 先为非法输入写测试，再让测试通过
- "修复 bug" → 先写复现测试，再让测试通过
- "重构 X" → 确保前后测试都通过

多步骤任务先给简短计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
```

强成功标准可独立推进，弱标准会不断需要澄清。

### 5. 注释

生成代码时，重要方法、公共接口、复杂逻辑和关键字段应添加简洁注释，说明设计意图和隐藏约束。不写与代码字面重复的无信息量注释。

### 6. Git 提交

提交信息用一句话描述本次提交的内容。若涉及修改的文件或主题过多，应提醒用户拆分为多次独立提交，避免单次提交混杂不相关的变更。

## 禁止行为

- 在指南中写入类设计、流程步骤、详细架构表格
- 将架构细节重复放在多个位置
- 在设计文档中写详细代码（源码即详细文档）
- 使用营销语气或过度热情的表达
- **未经用户明确许可，禁止自动执行写操作**（如修改文件、生成代码到磁盘等）
- **禁止自动执行 Git 推送**（`git push` 必须经用户明确确认后方可执行）

# Git 提交并推送

## 触发场景

- 用户说"提交并推送到远端"、"commit and push"
- 用户说"push 代码"、"提交代码"、"推送到远程"
- 用户说"提交到 xxx 分支"（指定分支时）

## 执行步骤

### 1. 检查工作区状态

```bash
git status
```

- 无变更 → 告知用户当前工作区干净，无需操作
- 有未暂存变更 → 进入步骤 2

### 2. 确定目标分支

| 用户指令 | 目标分支 |
|----------|----------|
| 未指定分支 | `develop` |
| "提交到 main" | `main` |
| "提交到 feature/xxx" | `feature/xxx` |

若当前不在目标分支，执行检出：
```bash
git checkout <branch>
```

### 3. 暂存与提交

```bash
git add -A
git commit -m "<message>"
```

**提交信息规则**：
- 用户未指定信息时，用一句话概括变更内容（如 "更新 xxx 配置"、"修复 yyy 问题"）
- 用户指定信息时，直接使用用户原话
- 若涉及文件或主题过多，提醒用户拆分提交，避免单次提交混杂不相关变更

### 4. 推送到远端

默认推送到**所有已配置的远端**：
```bash
git remote | xargs -I {} git push {} <branch>
```

或依次显式推送：
```bash
git push origin <branch>
git push github <branch>
```

### 5. 汇报结果

简洁输出：
- 提交哈希
- 提交信息
- 推送成功的远端列表

## 分支确认策略

| 目标分支 | 行为 |
|----------|------|
| `develop` | 直接执行，无需额外确认 |
| `main` | **暂停并提示用户**："目标分支为 main，确认继续合并/提交吗？" 用户确认后再执行 |
| 其他分支 | 直接执行，但备注当前操作的分支名 |

## 反模式

| 错误做法 | 正确做法 |
|----------|----------|
| 直接 push 到 main 不确认 | main 分支操作前必须二次确认 |
| 忽略未暂存文件 | 先 `git status` 检查，确保不遗漏 |
| 为每个文件单独 commit（除非用户要求） | 同一主题变更合并为单次提交 |
| 提交信息含糊（如 "update"） | 具体说明变更内容（如 "修复备份路径解析错误"） |
| 忘记 push tag（如发版场景） | 若涉及 tag，同时推送 `git push origin <tag>` |
